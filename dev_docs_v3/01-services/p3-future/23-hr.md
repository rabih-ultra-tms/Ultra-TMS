## 7. Business Rules

1. **Employee Types:** Employees are categorized by type: FULL_TIME, PART_TIME, CONTRACT, SEASONAL. Type determines benefits eligibility, PTO accrual rates, and overtime rules. CDL drivers have additional compliance requirements (medical card, license renewal tracking).

2. **Payroll Integration:** HR stores employee compensation data (salary/hourly rate, pay frequency, tax withholdings, deductions). Actual payroll processing is expected to integrate with an external payroll provider (e.g., ADP, Gusto). The system tracks pay periods and provides data exports but does not process direct deposits.

3. **Time Tracking:** Employees clock in/out via the time tracking module. Overtime is calculated per DOT/FMCSA rules for drivers and per state labor law for office staff. Weekly hours > 40 trigger overtime rate (1.5x). Time entries require manager approval before payroll export. Drivers have Hours of Service (HOS) constraints that must be validated.

4. **PTO Management:** PTO accrues based on employee type and tenure. Accrual rates: FULL_TIME = 1.25 days/month (15 days/year), increasing to 1.67 days/month after 3 years. PART_TIME = pro-rated. PTO requests follow an approval workflow: Employee submits -> Manager approves/denies -> HR notified. Requests cannot exceed available balance. Blackout dates may be configured per department.

5. **Department & Position Hierarchy:** Each employee belongs to one department and holds one position. Departments have a manager (an employee). Positions define pay grade ranges. Org chart is derived from department-manager relationships.

6. **Onboarding Workflow:** New employee onboarding includes: personal info collection, tax form completion (W-4/I-9), benefits enrollment, equipment assignment, training schedule, and policy acknowledgments. Onboarding status is tracked per step.

7. **Compliance & Certifications:** For TMS-specific roles, the system tracks CDL class/endorsements, medical card expiry, drug test dates, and safety training completion. Expiring certifications trigger alerts 30 days before expiry, similar to carrier insurance expiry logic.

8. **Soft Delete:** All HR records use soft delete. Terminated employees remain in the system for historical reporting and audit trail but are excluded from active headcount and payroll.

## 8. Data Model

### Employee
```
Employee {
  id              String (UUID)
  firstName       String
  lastName        String
  email           String (unique per tenant)
  phone           String?
  employeeNumber  String (unique per tenant)
  type            EmployeeType (FULL_TIME, PART_TIME, CONTRACT, SEASONAL)
  status          EmployeeStatus (ACTIVE, ON_LEAVE, TERMINATED, ONBOARDING)
  hireDate        DateTime
  terminationDate DateTime?
  departmentId    String (FK -> Department)
  positionId      String (FK -> Position)
  locationId      String (FK -> Location)
  managerId       String? (FK -> Employee, self-ref)
  salary          Decimal?
  hourlyRate      Decimal?
  payFrequency    PayFrequency? (WEEKLY, BIWEEKLY, SEMIMONTHLY, MONTHLY)
  tenantId        String
  createdAt       DateTime
  updatedAt       DateTime
  deletedAt       DateTime?
}
```

### Department
```
Department {
  id          String (UUID)
  name        String
  code        String (unique per tenant)
  managerId   String? (FK -> Employee)
  parentId    String? (FK -> Department, self-ref for hierarchy)
  employees   Employee[]
  tenantId    String
  createdAt   DateTime
  updatedAt   DateTime
  deletedAt   DateTime?
}
```

### Position
```
Position {
  id          String (UUID)
  title       String
  code        String (unique per tenant)
  payGradeMin Decimal?
  payGradeMax Decimal?
  departmentId String? (FK -> Department)
  employees   Employee[]
  tenantId    String
  createdAt   DateTime
  updatedAt   DateTime
  deletedAt   DateTime?
}
```

### Location
```
Location {
  id          String (UUID)
  name        String
  address     String
  city        String
  state       String
  zipCode     String
  employees   Employee[]
  tenantId    String
  createdAt   DateTime
  updatedAt   DateTime
  deletedAt   DateTime?
}
```

### TimeEntry
```
TimeEntry {
  id          String (UUID)
  employeeId  String (FK -> Employee)
  clockIn     DateTime
  clockOut    DateTime?
  breakMinutes Int (default: 0)
  totalHours  Decimal? (computed)
  isOvertime  Boolean (default: false)
  status      TimeEntryStatus (PENDING, APPROVED, REJECTED)
  approvedBy  String? (FK -> Employee)
  notes       String?
  tenantId    String
  createdAt   DateTime
  updatedAt   DateTime
}
```

### TimeOffRequest
```
TimeOffRequest {
  id          String (UUID)
  employeeId  String (FK -> Employee)
  type        TimeOffType (VACATION, SICK, PERSONAL, BEREAVEMENT, JURY_DUTY)
  startDate   DateTime
  endDate     DateTime
  totalDays   Decimal
  reason      String?
  status      TimeOffStatus (PENDING, APPROVED, DENIED, CANCELLED)
  reviewedBy  String? (FK -> Employee)
  reviewedAt  DateTime?
  reviewNotes String?
  tenantId    String
  createdAt   DateTime
  updatedAt   DateTime
}
```

## 9. Validation Rules

| Field | Rule | Error Message |
|-------|------|--------------|
| `email` | Valid email, unique per tenant | "Email already registered" |
| `employeeNumber` | Required, unique per tenant | "Employee number already in use" |
| `type` | IsEnum(EmployeeType) | "Invalid employee type" |
| `status` | IsEnum(EmployeeStatus) | "Invalid employee status" |
| `hireDate` | Required, valid date | "Hire date is required" |
| `departmentId` | Must reference existing Department | "Department not found" |
| `positionId` | Must reference existing Position | "Position not found" |
| `locationId` | Must reference existing Location | "Location not found" |
| `salary` or `hourlyRate` | At least one required, positive number | "Compensation is required" |
| TimeEntry `clockOut` | Must be after `clockIn` | "Clock out must be after clock in" |
| TimeEntry `clockIn` | No overlapping entries for same employee | "Overlapping time entry exists" |
| TimeOff `endDate` | Must be >= `startDate` | "End date must be on or after start date" |
| TimeOff `totalDays` | Must not exceed available PTO balance | "Insufficient PTO balance" |
| Department `code` | Required, unique per tenant, alphanumeric | "Department code already in use" |
| Position `payGradeMin` | Must be < `payGradeMax` if both provided | "Minimum pay grade must be less than maximum" |

## 10. Status States

### Employee Status Machine
```
ONBOARDING -> ACTIVE (onboarding steps completed)
ACTIVE -> ON_LEAVE (approved leave of absence)
ACTIVE -> TERMINATED (termination processed)
ON_LEAVE -> ACTIVE (return from leave)
ON_LEAVE -> TERMINATED (termination during leave)
TERMINATED -> (terminal state, no transitions out)
```

### Time Entry Status Machine
```
PENDING -> APPROVED (manager approval)
PENDING -> REJECTED (manager rejection with reason)
REJECTED -> PENDING (employee resubmits corrected entry)
APPROVED -> (terminal state for payroll export)
```

### Time Off Request Status Machine
```
PENDING -> APPROVED (manager approval)
PENDING -> DENIED (manager denial with reason)
PENDING -> CANCELLED (employee cancels own request)
APPROVED -> CANCELLED (employee cancels before start date)
DENIED -> (terminal state)
CANCELLED -> (terminal state)
```
