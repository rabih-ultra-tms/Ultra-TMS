## 7. Business Rules

1. **Employee Types:** Employees are categorized by type: FULL_TIME, PART_TIME, CONTRACT, TEMP. Type determines benefits eligibility, PTO accrual rates, and overtime rules. CDL drivers have additional compliance requirements (medical card, license renewal tracking).

2. **Employee Auto-Numbering:** Employee numbers are auto-generated in `EMP-XXXXX` format (auto-incrementing). Numbers are globally unique via `@unique` constraint.

3. **Circular Hierarchy Detection:** `validateManager()` walks the manager chain to prevent circular reporting relationships before saving.

4. **PTO Balance Management:** TimeOffBalanceService tracks per-employee, per-type, per-year balances with `@@unique([employeeId, timeOffType, year])`. Implements addPending/removePending/movePendingToUsed pattern. Note: `movePendingToUsed()` is currently a stub (returns `{ moved: true }` without actual balance transfer logic).

5. **Time Off Approval Workflow:** Employee submits -> Manager approves/denies -> balance updated. Requests cannot exceed available balance. Approval/denial fires EventEmitter events.

6. **EventEmitter Events:** 6 events: `employee.created`, `employee.updated`, `employee.terminated`, `timeoff.requested`, `timeoff.approved`, `timeoff.denied`.

7. **Department & Position Hierarchy:** Each employee belongs to one department and holds one position. Departments have a manager (an employee). Positions define pay grade ranges. Org chart is derived from department-manager relationships.

8. **Soft Delete:** Employee correctly uses soft delete (`deletedAt`). **BUG:** Departments, Positions, and Locations use hard delete despite having `deletedAt` fields in Prisma.

9. **Payroll Integration:** HR stores employee compensation data (annualSalary, hourlyRate). Actual payroll processing is expected to integrate with an external payroll provider (e.g., ADP, Gusto). Not started.

10. **Onboarding Models:** OnboardingChecklist and OnboardingTask Prisma models exist but have no API controllers — future capability.

## 9. Validation Rules

| Field | Rule | Error Message |
|-------|------|--------------|
| `email` | Valid email, unique per tenant | "Email already registered" |
| `employeeNumber` | Auto-generated `EMP-XXXXX`, globally unique | "Employee number already in use" |
| `type` | IsEnum(EmploymentType) | "Invalid employee type" |
| `status` | IsEnum(EmploymentStatus) | "Invalid employee status" |
| `hireDate` | Required, valid date | "Hire date is required" |
| `departmentId` | Must reference existing Department | "Department not found" |
| `positionId` | Must reference existing Position | "Position not found" |
| `locationId` | Must reference existing Location | "Location not found" |
| `managerId` | Circular hierarchy check via validateManager() | "Circular reporting relationship detected" |
| `annualSalary` or `hourlyRate` | Positive number if provided | "Compensation must be positive" |
| TimeOff `endDate` | Must be >= `startDate` | "End date must be on or after start date" |
| TimeOff `totalDays` | Must not exceed available PTO balance | "Insufficient PTO balance" |
| Department `code` | Required, unique per tenant, alphanumeric | "Department code already in use" |
| Position `payGradeMin` | Must be < `payGradeMax` if both provided | "Minimum pay grade must be less than maximum" |

## 10. Status States

### Employee Status Machine
```
ACTIVE -> ON_LEAVE (approved leave of absence)
ACTIVE -> TERMINATED (termination processed via POST /:id/terminate)
ON_LEAVE -> ACTIVE (return from leave)
ON_LEAVE -> TERMINATED (termination during leave)
TERMINATED -> (terminal state, no transitions out)
```

Note: No ONBOARDING status exists in Prisma. Only 3 states: ACTIVE, ON_LEAVE, TERMINATED.

### Time Off Request Status Machine
```
PENDING -> APPROVED (manager approval via POST /:id/approve)
PENDING -> DENIED (manager denial via POST /:id/deny)
PENDING -> CANCELLED (employee cancels own request)
APPROVED -> CANCELLED (employee cancels before start date)
DENIED -> (terminal state)
CANCELLED -> (terminal state)
```
