# Workflow Module API Spec

**Module:** `apps/api/src/modules/workflow/`
**Base path:** `/api/v1/`
**Controllers:** WorkflowsController, TemplatesController, ApprovalsController, ExecutionsController, WorkflowExecutionsController
**Auth:** `JwtAuthGuard` + `RolesGuard`
**Scope:** P2

---

## WorkflowsController

**Route prefix:** `workflows`

| Method | Path | Description | Roles |
|--------|------|-------------|-------|
| GET | `/workflows/actions` | List available workflow actions | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| GET | `/workflows/actions/:type` | Get action details by type | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| GET | `/workflows/triggers` | List available workflow triggers | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| GET | `/workflows/triggers/:type` | Get trigger details by type | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| POST | `/workflows/validate` | Validate workflow definition | ADMIN, OPERATIONS_MANAGER |
| GET | `/workflows` | List workflows | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| POST | `/workflows` | Create workflow | ADMIN, OPERATIONS_MANAGER |
| GET | `/workflows/:id` | Get workflow by ID | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |
| PUT | `/workflows/:id` | Update workflow | ADMIN, OPERATIONS_MANAGER |
| DELETE | `/workflows/:id` | Delete workflow (204 No Content) | ADMIN |
| POST | `/workflows/:id/publish` | Publish workflow | ADMIN |
| POST | `/workflows/:id/activate` | Activate workflow | ADMIN |
| POST | `/workflows/:id/deactivate` | Deactivate workflow | ADMIN |
| POST | `/workflows/:id/execute` | Execute workflow manually | ADMIN, OPERATIONS_MANAGER, DISPATCHER |
| GET | `/workflows/:id/stats` | Get workflow execution stats | ADMIN, OPERATIONS_MANAGER, DISPATCHER, SALES_REP |

**DTOs:** CreateWorkflowDto, UpdateWorkflowDto, ExecuteWorkflowDto, WorkflowQueryDto

**Notes:**
- Actions and triggers are served from `actions.constants.ts` (static action library and trigger schemas)
- Workflow lifecycle: Draft -> Published -> Active/Inactive
- Only ADMIN can delete, publish, activate, or deactivate
- Validate endpoint allows pre-checking definitions before saving

## TemplatesController

**Route prefix:** `workflow-templates`

| Method | Path | Description | Roles |
|--------|------|-------------|-------|
| GET | `/workflow-templates` | List templates (optional `category` filter) | ADMIN, OPERATIONS_MANAGER |
| GET | `/workflow-templates/:id` | Get template by ID | ADMIN, OPERATIONS_MANAGER |
| POST | `/workflow-templates` | Create template | ADMIN |
| PUT | `/workflow-templates/:id` | Update template | ADMIN |
| DELETE | `/workflow-templates/:id` | Delete template (204 No Content) | ADMIN |
| POST | `/workflow-templates/:id/create-workflow` | Create workflow from template | ADMIN, OPERATIONS_MANAGER |

**DTOs:** CreateTemplateDto, UpdateTemplateDto, CreateFromTemplateDto

**Notes:**
- Templates are pre-built workflow definitions that can be instantiated
- Only ADMIN can create/update/delete templates
- OPERATIONS_MANAGER can browse and create workflows from templates

## ApprovalsController

**Route prefix:** `approvals`

| Method | Path | Description | Roles |
|--------|------|-------------|-------|
| GET | `/approvals` | List all approvals | VIEWER, USER, MANAGER, ADMIN |
| GET | `/approvals/pending` | List pending approvals for current user | VIEWER, USER, MANAGER, ADMIN |
| GET | `/approvals/stats` | Get approvals statistics | VIEWER, USER, MANAGER, ADMIN |
| GET | `/approvals/:id` | Get approval by ID | VIEWER, USER, MANAGER, ADMIN |
| POST | `/approvals/:id/approve` | Approve request (200 OK) | MANAGER, ADMIN |
| POST | `/approvals/:id/reject` | Reject request (200 OK) | MANAGER, ADMIN |
| POST | `/approvals/:id/delegate` | Delegate to another user (200 OK) | USER, MANAGER, ADMIN |

**DTOs:** ApprovalQueryDto, ApprovalDecisionDto, RejectApprovalDto, DelegateApprovalDto

**Notes:**
- Approval requests are generated by workflow actions
- Pending endpoint filters by assigned approver (current user)
- Stats provide counts by status (pending, approved, rejected)
- Delegation allows passing approval responsibility to another user
- Uses `@HttpCode(HttpStatus.OK)` on action endpoints (override default 201)

## Architecture Notes

- **5 controllers** with 35 endpoints total
- Workflow engine supports: triggers (event, schedule, manual), conditions, and actions
- Action library includes: send_email, create_task, update_status, assign_carrier, notify_user, webhook, etc.
- Trigger types: EVENT (entity changes), SCHEDULE (cron), MANUAL (user-initiated)
- Execution model: each run produces steps and logs for audit trail
- Cancel and retry provide execution recovery
- Two separate controllers share the same `ExecutionsService` (flat route + nested route)
