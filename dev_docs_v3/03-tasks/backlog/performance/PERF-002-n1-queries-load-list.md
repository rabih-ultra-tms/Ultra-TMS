# PERF-002: Fix N+1 Queries in Load List Endpoint

**Priority:** P1
**Service:** TMS Core
**Scope:** Optimize the load list endpoint to avoid N+1 query patterns

## Current State
The load list endpoint in `apps/api/src/modules/tms/loads.service.ts` uses `findMany` with multiple `include` levels. The service includes `order` (with nested `customer`), `carrier`, `stops`, and other relations. While Prisma `include` batches some queries, nested includes can still generate excessive SQL.

Evidence from `loads.service.ts`: the `findMany` call includes `carrier: true`, `order: { include: { customer: true } }`, and `stops: true` in the list query, which generates JOIN queries that may be inefficient for large datasets.

## Requirements
- Analyze actual SQL generated by the list query using Prisma query logging
- Optimize includes: only include fields needed for list view (not full detail)
- Consider separate `select` for list vs detail endpoints
- Add database indexes if missing on frequently filtered/sorted columns
- Target: list page query under 100ms for 1000+ loads

## Acceptance Criteria
- [ ] SQL query count analyzed and documented
- [ ] List endpoint uses optimized select/include
- [ ] Unnecessary nested includes removed from list query
- [ ] Response time under 100ms for 25-load page
- [ ] No change to API response shape for frontend consumers
- [ ] Missing indexes identified and added
